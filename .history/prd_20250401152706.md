# Pokémon State Management Comparison App - PRD

## 1. Goal

To build a React application that demonstrates and compares three different approaches to fetching and managing server state using the public PokéAPI:

1.  **React Query:** Highlighting its caching, background refetching, and declarative data fetching capabilities.
2.  **RTK Query:** Showcasing Redux Toolkit's integrated data fetching and caching solution.
3.  **Manual Redux + Fetch:** Illustrating a more traditional approach using basic Redux state management combined with manual fetching in `useEffect`, mimicking a common pattern that often leads to redundant fetching and boilerplate.

The goal is to provide a clear, practical example for developers comparing these libraries, especially highlighting the benefits of dedicated server state libraries like React Query and RTK Query over manual implementations.

## 2. Key Features & Requirements

*   **Framework/Tooling:** React (Vite + TypeScript).
*   **UI:** Material UI (MUI) for components and styling. Minimal/functional styling is sufficient.
*   **Routing:** `react-router-dom` for navigation between the three example sections and detail views.
*   **API:** PokéAPI v2 (`https://pokeapi.co/api/v2/`).
*   **Core UI Components (Shared):**
    *   `PokemonCard`: Displays basic info (name, image, type). Fetches its own details dependently. Shows skeleton loading state. Links to detail page.
    *   `PokemonGrid`: Displays a grid of `PokemonCard` components for the first 151 Pokémon. Shows overall list loading/error state.
    *   `PokemonDetail`: Displays more comprehensive details for a selected Pokémon (stats, abilities, etc.). Shows loading/error state.
    *   `Layout`: Contains navigation (AppBar/Tabs) to switch between the three versions and renders the main content area.
    *   Loading/Error Components: Reusable MUI `Skeleton` components and simple error message displays/fallbacks.
*   **Three Implementations (Separate Routes):**
    *   **/react-query:** Grid + Detail views powered by `@tanstack/react-query`. Includes a cache invalidation button. Demonstrates efficient caching on navigation.
    *   **/rtk-query:** Grid + Detail views powered by RTK Query (`@reduxjs/toolkit/query/react`). Includes a cache tag invalidation button. Demonstrates efficient caching integrated with Redux.
    *   **/redux:** Grid + Detail views powered by manual Redux (`@reduxjs/toolkit`, `react-redux`, `createAsyncThunk`) and `fetch`/`axios` within `useEffect`. *No* smart caching between navigations (refetches on mount). Cache bust button is disabled with a tooltip.
*   **Data Fetching:**
    *   Fetch initial list of 151 Pokémon (`/pokemon?limit=151`).
    *   Fetch individual Pokémon details (by name/URL) for cards and detail pages. This demonstrates dependent queries.
*   **State Representation:** Clearly show loading states (skeletons) and error states for both list and detail fetching.

## 4. Implementation Plan

### Phase 1: Project Setup & Core UI

1.  **Initialize Project:** `npm create vite@latest pokemon-server-state-example -- --template react-ts`.
2.  **Install Dependencies:** `react-router-dom`, `@reduxjs/toolkit`, `react-redux`, `@tanstack/react-query`, `@mui/material`, `@emotion/react`, `@emotion/styled`, `axios` (optional).
3.  **Basic Structure:** Set up folders: `src/components`, `src/pages`, `src/services`, `src/store`, `src/hooks`, `src/types`.
4.  **API Service (`src/services/pokeapi.ts`):** Create fetch/axios functions (`getPokemonList`, `getPokemonDetails`) and define API response types.
5.  **Core UI Components (`src/components`):** Create `PokemonCard`, `PokemonGrid`, `PokemonDetail`, `Layout`, `LoadingSkeleton`, `ErrorDisplay`. Implement structure and styling with MUI, placeholders for logic.
6.  **Routing (`src/App.tsx`):** Set up `BrowserRouter` and routes for `/`, `/react-query`, `/react-query/:pokemonName`, `/rtk-query`, `/rtk-query/:pokemonName`, `/redux`, `/redux/:pokemonName` using the `Layout`.

### Phase 2: React Query Implementation

1.  **Provider:** Wrap app in `QueryClientProvider` (`src/main.tsx`).
2.  **Page (`src/pages/ReactQueryPage.tsx`):** Use `useQuery` to fetch list, pass state to `PokemonGrid`, implement cache bust button (`invalidateQueries`).
3.  **Card Integration (`PokemonCard`):** Use `useQuery` to fetch details, handle loading/error states, link to detail page.
4.  **Detail Page (`PokemonDetail` - RQ Version):** Use `useParams` and `useQuery` to fetch and display details.

### Phase 3: RTK Query Implementation

1.  **Redux Store (`src/store/store.ts`):** Configure store with RTK Query middleware.
2.  **API Slice (`src/store/pokemonApiSlice.ts`):** Define API endpoints using `createApi`, configure tags for caching.
3.  **Provider:** Wrap app in Redux `Provider` (`src/main.tsx`).
4.  **Page (`src/pages/RtkQueryPage.tsx`):** Use generated `useGetPokemonListQuery` hook, pass state to `PokemonGrid`, implement cache bust button (`invalidateTags`).
5.  **Card Integration (`PokemonCard`):** Use generated `useGetPokemonDetailsQuery` hook, handle loading/error states, link to detail page.
6.  **Detail Page (`PokemonDetail` - RTKQ Version):** Use `useParams` and generated hook.

### Phase 4: "Pure" Redux Implementation

1.  **State Slice (`src/store/pokemonSlice.ts`):** Define state shape, use `createSlice` for reducers, `createAsyncThunk` for `fetchPokemonList` and `fetchPokemonDetails`.
2.  **Page (`src/pages/ReduxPage.tsx`):** Use `useSelector`, `useDispatch`, fetch list in `useEffect`, pass state to `PokemonGrid`, add disabled cache bust button.
3.  **Card Integration (`PokemonCard`):** Use `useSelector`, `useDispatch`, fetch details in `useEffect`, handle loading/error states, link to detail page. Emphasize refetch-on-mount behavior.
4.  **Detail Page (`PokemonDetail` - Redux Version):** Use `useParams`, `useSelector`, `useDispatch`, fetch details in `useEffect`. Emphasize refetch-on-mount behavior.

### Phase 5: Refinement & Review

1.  **Styling:** Ensure consistency in UI, loading, and error states across all versions.
2.  **Testing:** Manually verify functionality, caching behavior, and expected differences between the implementations.
3.  **README:** Create `README.md` explaining the project, setup, and observed differences.

## 5. Non-Goals

*   User authentication or accounts.
*   Complex state mutations (e.g., "catching" Pokémon).
*   Offline support or persistence beyond session cache.
*   Server-Side Rendering (SSR) or advanced optimization techniques.
*   Unit/Integration tests (focus is on demonstrating concepts).
